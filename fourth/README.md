# ЛР 3.5: формирование изображения методом обратной трассировки лучей

## Цель работы

Сформировать изображение трехмерной сцены методом обратной трассировки лучей с учетом локального освещения, зеркальной составляющей и теней. Для поиска пересечений лучей с треугольной сеткой используется Intel Embree 3.

В текущей реализации выполняется одно отражательное вычисление в точке пересечения: первичный луч камеры ищет ближайшую поверхность, после чего для этой поверхности рассчитывается прямое локальное освещение от точечных источников. Полноценное многократное распространение лучей между поверхностями не используется.

## Используемая технология

Проект реализован на C++17 и использует Intel Embree 3:

- `RTCDevice` — устройство Embree;
- `RTCScene` — ускоряющая структура сцены;
- `RTCGeometry` — треугольная геометрия;
- `rtcIntersect1` — поиск ближайшего пересечения первичного луча;
- `rtcOccluded1` — проверка затенения лучом к источнику света.

Embree отвечает только за геометрическую часть трассировки. Расчет камеры, материалов, освещения, теней, гамма-коррекции, CSV-вывода и PPM-изображения выполняется в коде лабораторной.

## Сцена

Сцена задается процедурно в `createDefaultScene()` без загрузки OBJ. Это сделано намеренно: простая контролируемая сцена легче проверяется при защите и позволяет явно показать работу первичных лучей, теней и локального освещения.

Состав сцены:

- пол из двух треугольников;
- задняя стена из двух треугольников;
- левая боковая стена из двух треугольников;
- пирамида из шести треугольников;
- два цветных точечных источника света;
- камера с разрешением `640x640`.

Такой набор достаточен для демонстрации:

- пересечения лучей с треугольной сеткой;
- видимых и невидимых участков сцены;
- теней через `rtcOccluded1`;
- диффузной составляющей;
- зеркальной составляющей;
- разных материалов на разных мешах.

## Модель камеры

Для каждого пикселя строится первичный луч из положения камеры через плоскость изображения. Вектор направления рассчитывается по положению камеры, целевой точке, верхнему направлению `up`, вертикальному углу обзора и соотношению сторон изображения.

Условно:

```text
ray.origin = camera.position
ray.direction = normalize(forward + right * screen_x + camera_up * screen_y)
```

где:

- `forward` — направление от камеры к `target`;
- `right` — правый вектор камеры;
- `camera_up` — уточненный верхний вектор камеры;
- `screen_x`, `screen_y` — координаты пикселя на плоскости изображения.

## Модель освещения

Для найденной точки пересечения рассчитывается локальное освещение от каждого источника света.

### Ослабление света расстоянием

Для источника света используется обратно-квадратичное ослабление:

$$
L_i = \frac{I_i}{r_i^2}
$$

где:

- $I_i$ — RGB-интенсивность источника;
- $r_i$ — расстояние от точки поверхности до источника.

### Диффузная составляющая

Используется модель Ламберта:

$$
C_d = L_i \cdot K_d \cdot k_d \cdot \max(0, \vec{n} \cdot \vec{l})
$$

где:

- $K_d$ — диффузный RGB-цвет материала;
- $k_d$ — коэффициент диффузного отражения;
- $\vec{n}$ — нормаль поверхности;
- $\vec{l}$ — направление от точки поверхности к источнику света.

### Зеркальная составляющая

Используется модель Блинна-Фонга:

$$
\vec{h} = \frac{\vec{l} + \vec{v}}{\|\vec{l} + \vec{v}\|}
$$

$$
C_s = L_i \cdot K_s \cdot k_s \cdot \max(0, \vec{n} \cdot \vec{h})^p
$$

где:

- $K_s$ — зеркальный RGB-цвет материала;
- $k_s$ — коэффициент зеркального отражения;
- $p$ — показатель блеска `shininess`;
- $\vec{v}$ — направление от поверхности к камере;
- $\vec{h}$ — half-vector.

### Проверка теней

Для каждого источника строится теневой луч:

```text
shadow_ray.origin = hit_position + normal * shadow_bias
shadow_ray.direction = direction_to_light
```

Если `rtcOccluded1` находит пересечение между точкой поверхности и источником, вклад этого источника не добавляется.

## Алгоритм работы программы

1. Создать учебную сцену из треугольников.
2. Создать `RTCDevice` и `RTCScene`.
3. Загрузить все треугольные меши в Embree как `RTC_GEOMETRY_TYPE_TRIANGLE`.
4. Для каждого пикселя изображения построить первичный луч камеры.
5. Найти ближайшее пересечение через `rtcIntersect1`.
6. Если пересечения нет, записать цвет фона.
7. Если пересечение есть, вычислить точку, нормаль и материал.
8. Для каждого источника света проверить ориентацию поверхности и построить теневой луч.
9. Через `rtcOccluded1` определить, находится ли точка в тени.
10. Если источник видим, добавить диффузную и зеркальную составляющие.
11. Ограничить RGB-значения диапазоном `0..1`.
12. Сохранить изображение в `render.ppm` с гамма-коррекцией.
13. Сохранить входные данные и сводные результаты в CSV-файлы.
14. Вывести таблицы в консоль на русском языке.

## Выходные файлы

После запуска в директории `fourth/output` создаются:

| Файл | Содержимое |
|------|------------|
| `render.ppm` | Итоговое изображение сцены в формате PPM `P3` |
| `input_camera.csv` | Положение камеры, целевая точка, `up`, угол обзора и разрешение |
| `input_lights.csv` | Положения и RGB-интенсивности точечных источников света |
| `input_triangles.csv` | Геометрия треугольников, нормали, материалы и коэффициенты отражения |
| `render_summary.csv` | Разрешение, количество треугольников, источников, первичных лучей, попаданий, теневых лучей и яркостная статистика |
| `control_rays.csv` | Контрольные лучи для нескольких пикселей: факт пересечения, расстояние, геометрия, нормаль, цвет и пояснение |

Все CSV-файлы сохраняются с разделителем `;`, как и в предыдущих лабораторных.

## Контрольные лучи

Для проверки корректности дополнительно сохраняется таблица `control_rays.csv`. В ней фиксируются несколько характерных пикселей изображения:

- центральный пиксель;
- верхняя левая область;
- верхняя правая область;
- нижняя левая область;
- нижняя правая область.

Для каждого контрольного луча сохраняются:

- координаты пикселя;
- наличие пересечения;
- расстояние до пересечения;
- идентификаторы геометрии и примитива Embree;
- точка пересечения;
- нормаль;
- рассчитанный цвет;
- пояснение результата.

Эта таблица используется как численное доказательство того, что трассировщик действительно строит лучи, находит пересечения и применяет локальное освещение.

## Требование к Intel Embree 3

Проект ожидает установленный Intel Embree 3.

В `CMakeLists.txt` сначала используется:

```cmake
find_package(embree 3 CONFIG QUIET)
```

Если пакетная конфигурация не найдена, выполняется ручной поиск:

- заголовков `embree3/rtcore.h` и `embree3/rtcore_ray.h`;
- библиотеки `embree3` или `embree3_d`;
- путей из переменных окружения `EMBREE_DIR` и `EMBREE_ROOT`.

Если Embree не найден, CMake завершает конфигурацию с понятным сообщением об ошибке.

## Ограничения текущей реализации

- Сцена задается вручную в коде, OBJ-загрузка пока не используется.
- Реализовано прямое локальное освещение, а не полноценный многократный path tracing.
- Антиалиасинг не используется: один первичный луч на пиксель.
- Прозрачность и преломление не реализованы.
- Материалы задаются простыми коэффициентами Ламберта и Блинна-Фонга.

Эти ограничения не мешают выполнить базовую цель ЛР 3.5: построить изображение треугольной сцены методом обратной трассировки лучей с локальным освещением и тенями.

## Краткий вывод

Реализация формирует изображение трехмерной сцены из треугольников, использует Intel Embree 3 для поиска пересечений, вычисляет локальное RGB-освещение по моделям Ламберта и Блинна-Фонга, проверяет тени через `rtcOccluded1`, сохраняет результат в PPM и формирует CSV-таблицы, пригодные для отчета.
