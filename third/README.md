# ЛР №3, часть 1

## Цель работы

Реализовать расчет RGB-яркости в заданной точке зеркально-диффузной поверхности для нескольких направлений наблюдения. В этой части работы не строится изображение: вычисляется только яркость в одной точке поверхности для набора положений наблюдателя.

## Краткая физическая постановка

Рассматривается одна точка поверхности `P_T`, для которой заданы:
- положение точки в пространстве;
- нормаль к поверхности `N`;
- положение источника света и его RGB-интенсивность;
- параметры материала поверхности;
- несколько положений наблюдателя.

Для каждого положения наблюдателя вычисляются направление на источник света, направление на наблюдателя, отраженное направление падающего луча и итоговая RGB-яркость как сумма диффузной и зеркальной составляющих.

Дополнительно рассматриваются контрольные и пограничные сценарии, в которых яркость должна быть нулевой или заметно уменьшаться из-за геометрии сцены. Это нужно, чтобы показать корректность работы программы на односторонней поверхности.

## Используемые формулы

### Отраженный луч

```text
O' = O - 2 * (O dot N) * N
```

Где:
- `O` - падающий луч;
- `N` - нормаль к поверхности;
- `O'` - отраженный луч.

### Локальная модель яркости

```text
L = L_diffuse + L_specular
L_diffuse = I * Kd_color * kd * max(0, N dot L_dir)
L_specular = I * Ks_color * ks * max(0, R dot V)^shininess
```

Где:
- `I` - RGB-интенсивность источника света;
- `Kd_color` - диффузный цвет материала;
- `Ks_color` - цвет зеркальной составляющей;
- `kd` - коэффициент диффузного отражения;
- `ks` - коэффициент зеркального отражения;
- `N` - нормаль поверхности;
- `L_dir` - направление от точки поверхности к источнику;
- `R` - отраженное направление падающего света;
- `V` - направление от точки поверхности к наблюдателю.

Для данной ЛР поверхность считается односторонней: если источник света или наблюдатель находятся с обратной стороны поверхности, вклад в яркость принимается равным нулю.

## Описание входных данных

Для хранения сцены используются dataclass-структуры:
- `LightSource` - `position: Vec3`, `intensity: Vec3`;
- `SurfacePoint` - `position: Vec3`, `normal: Vec3`;
- `Material` - `diffuse_color: Vec3`, `specular_color: Vec3`, `kd: float`, `ks: float`, `shininess: float`;
- `SceneInput` - объединяет `light`, `surface_point`, `material`, `observer_positions: list[Vec3]`.

Функция `create_default_scene()` создает осмысленный пример:
- точка поверхности находится в начале координат;
- нормаль направлена вверх вдоль оси `Z`;
- источник света расположен сбоку и сверху;
- заданы 5 положений наблюдателя;
- параметры материала выбраны простыми и удобными для объяснения на защите.

## Алгоритм работы

1. Создать входную сцену через `create_default_scene()`.
2. Сформировать входные таблицы для источника света, поверхности и положений наблюдателей.
3. Для каждого положения наблюдателя вычислить направление `L_dir` от точки поверхности к источнику.
4. Вычислить направление `V` от точки поверхности к наблюдателю.
5. Построить отраженный вектор `R` по формуле зеркального отражения.
6. Проверить, что источник и наблюдатель находятся на лицевой стороне поверхности.
7. Вычислить диффузную составляющую `L_diffuse`.
8. Вычислить зеркальную составляющую `L_specular`.
9. Получить итоговую RGB-яркость `L = L_diffuse + L_specular`.
10. Найти максимальную компоненту среди всех рассчитанных яркостей и построить нормализованные значения в диапазоне `0..1`.
11. Построить отдельный набор контрольных сценариев.
12. Проверить для каждого контрольного сценария, соответствует ли полученная яркость ожидаемому поведению.
13. Вывести таблицы в консоль и сохранить результаты в CSV-файлы.

## Нормализация яркости

Основной расчет выполняется в абсолютных или условных яркостях, то есть в значениях, которые напрямую получаются из формул локальной модели.

Для удобного сравнения результатов между разными направлениями наблюдения дополнительно выполняется нормализация:
- ищется максимальная компонента среди всех рассчитанных RGB-яркостей;
- каждое значение делится на этот максимум;
- если максимум равен нулю, нормализованные значения остаются нулевыми.

Нормализация нужна только для представления результатов в относительном виде. Gamma correction не используется, потому что в этой части ЛР изображение еще не формируется.

## Контрольные и пограничные сценарии

Для дополнительной проверки корректности расчета формируется отдельная таблица `third/control_scenarios.csv`.

В нее включены сценарии:
- `mirror_valid_reflection` - корректное зеркальное отражение, ожидается ненулевая яркость;
- `observer_below_mirror` - наблюдатель под зеркальной поверхностью, ожидается нулевая яркость;
- `light_below_surface` - источник под поверхностью, ожидается нулевая яркость;
- `diffuse_front_lit` - диффузная поверхность освещена спереди, ожидается ненулевая яркость;
- `diffuse_back_lit` - диффузная поверхность освещена с обратной стороны, ожидается нулевая яркость;
- `observer_back_side` - наблюдатель находится с обратной стороны односторонней поверхности, ожидается нулевая яркость.

Для поля `passed` используется допуск `EPS = 1e-9`:
- если ожидается нулевая яркость, проверяется, что все компоненты меньше `EPS`;
- если ожидается ненулевая яркость, проверяется, что хотя бы одна компонента больше `EPS`.

## Инструкция запуска

Из корня репозитория:

```bash
python third/main.py
```

При запуске программа:
- создает сцену по умолчанию;
- вычисляет яркость для пяти положений наблюдателя;
- вычисляет отдельные контрольные и пограничные сценарии;
- выводит таблицы в консоль;
- сохраняет результаты в CSV-файлы прямо в директорию `third`.

## Генерируемые CSV-файлы

После запуска создаются:
- `third/input_light_source.csv`
- `third/input_surface.csv`
- `third/input_observers.csv`
- `third/brightness_results.csv`
- `third/control_scenarios.csv`

Все CSV-файлы сохраняются с разделителем `;`.

## Краткий вывод

В результате реализован учебный Python-проект для первой части ЛР №3, который вычисляет яркость в заданной точке поверхности для нескольких направлений наблюдения по локальной модели освещения. Отдельный набор контрольных и пограничных сценариев показывает, что программа корректно обрабатывает случаи с нулевой яркостью при освещении или наблюдении с обратной стороны поверхности. Проект не выполняет рендеринг изображения, не использует трассировку сцены, не работает с OBJ и PPM, а решает именно задачу ЛР №3-1: расчет яркости в заданных направлениях наблюдения с сохранением результатов в CSV-таблицы.
